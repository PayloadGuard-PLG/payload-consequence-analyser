What PLI accepts as input from a diff:
PLI currently takes (user_text, model_text) pairs. For PLG integration, the mapping is:
PLG Context
PLI Input Mapping
PR description vs actual diff
user_text = PR title + body, model_text = diff summary
Old function vs new function
user_text = old_code, model_text = new_code
Commit message vs commit content
user_text = commit message, model_text = diff content
Comment justification vs change
user_text = inline PR comment, model_text = code it refers to
L4 (structural/AST) already gives you the which — which functions were deleted/modified. PLI tells you the what — what the modification actually means semantically.

Where it sits:
Between L4 (structural) and L3 (consequence). L4 tells PLI which functions changed. PLI analyzes what those changes mean. L3 scores the combined signals.

What it does NOT address:
Runtime behavior (that's L5c/eBPF's job)
INC-3 (direct push to main) — PLI analyzes content, not git workflow
Binary file analysis — PLI is text-only
PLI is the semantic layer PLG is missing. Z3 proves the scoring model is sound. PLI tells the scoring model what to score.
